
## 幽灵断流排查实录：Clash Verge、Windows 原生拨号与虚拟网卡的博弈

原来如此，这个结局确实提供了一个非常经典的技术视角。这说明 Windows 原生的 PPPoE 拨号组件在与现代网络代理架构的底层兼容上存在难以调和的结构性缺陷。

结合这个最终的真相，以下是为你整理的完整排查心路历程与技术总结博客。

---

### 问题的起点：诡异的“半断网”状态

在电脑端运行 Clash Verge（最新版），遭遇了经典的“代理失效”难题。现象充满矛盾且令人费解：

* **代理软件内置的连通性检测全绿**，同网络下手机端测试也完全正常。
* **电脑端浏览器（Edge/Zen）检测显示真实本地 IP**，且 Clash 面板毫无网页连接记录。
* **尝试开启 TUN 虚拟网卡模式**，不仅无法上网，连软件内置的 IP 检测也随之失效。

这种系统代理无法接管流量、TUN 模式直接导致路由黑洞的现象，开启了一场深入 Windows 底层网络协议栈的排查。

---

### 第一阶段：应用层协议与底层 DNS 的迷雾

起初，针对 TUN 模式下全局断流但 ping 域名延迟极低（<1ms）的现象，确认了 **Fake-IP 机制** 仍在拦截 DNS，但 TCP 流量在底层被截断。

* **禁用了浏览器的 QUIC（实验性 UDP）协议**，排除了 UDP 穿透导致直连漏网的干扰。
* **在 Clash 中关闭了 IPv6 DNS 解析**。此操作成功解除了 Windows 底层的 DNS/IPv6 路由死锁，使得国内网站（百度）恢复访问。

此时，国内直连打通，但境外流量依然如泥牛入海，没有产生任何超时或拒绝连接的日志。

---

### 第二阶段：PPPoE 拨号环境与路由接管断层

在确认网络环境为“校园网 PPPoE 宽带拨号”后，排查重心转移到网络链路本身的特殊限制上。

* 推测 PPPoE 特有的 8 字节头部导致 **MTU 超限**，产生大数据包静默丢弃，但尝试降低 Windows 拨号网卡的 MTU 未果。
* 鉴于节点均为 UDP 协议，推测校园网拨号网关对 UDP 代理实施了精准封锁。
* **关键对照测试**：断开电脑原生拨号，使用手机连接校园网并共享热点给电脑，代理瞬间完美运行。

这一结果直接排除了校园网物理链路的硬性阻断，将问题精准锁定在 **Windows 系统的原生 PPPoE 拨号逻辑** 与 **Clash 路由接管机制** 之间的冲突。

---

### 第三阶段：系统代理的隔离与 PAC 模式的突围

Windows 系统的“局域网代理设置”在面对 PPPoE 拨号连接时，存在严重的下发断层。浏览器在拨号状态下往往判定自身处于“无代理”环境，无视本地 Clash 端口，直接向物理网络发起请求。

为了验证这种操作系统级别的应用层隔离，在 Zen 浏览器内部手动配置了 **PAC（代理自动配置）模式**。网络瞬间恢复通畅。这证明了 Clash 核心的转发能力与节点链路完全健康，核心矛盾在于：**Windows 原生拨号网络无法将系统代理环境变量或 TUN 路由表正确作用于上层应用。**

---

### 第四阶段：意料之外的终局——虚拟网卡的降维打击

排查的最终转折干净利落：彻底解决问题的并非是对 Windows 原生拨号底层参数的繁琐修复，而是直接将其弃用。

通过**删除系统原有的 PPPoE 拨号连接**，转而使用第三方自动拨号软件创建的**“虚拟网卡”**进行网络连接，所有的代理异常瞬间消失。

其背后的技术逻辑十分清晰：

1. **协议栈隔离**：Windows 原生的 PPPoE 拨号在底层建立的是一个点对点逻辑适配器（PPP），其默认网关生成机制和代理环境变量继承极其特殊。当 Clash 尝试通过 TUN 模式或系统代理去劫持流量时，往往会被这套陈旧的协议栈架空，导致流量迷失在物理网卡与拨号网卡之间。
2. **标准化接口**：第三方拨号软件生成的专属虚拟网卡，向操作系统提供了一个标准化的底层接口。它屏蔽了 PPPoE 封装的复杂性，使 Windows 能够像处理普通以太网连接一样，为其分配标准的路由表并正常下发系统代理配置。Clash Verge 的接管机制也因此能够在这个标准化的逻辑网卡上稳定生效。

---

### 总结

此次排查完整揭露了 Windows 复杂网络栈中的一个典型盲区：**原生宽带拨号协议与现代代理路由接管机制的底层结构性不兼容**。最终的解决方案证明，在面对操作系统底层协议带来的死锁时，通过引入标准化的虚拟网卡层来重构网络拓扑，是跨越系统环境限制最直接有效的手段。
